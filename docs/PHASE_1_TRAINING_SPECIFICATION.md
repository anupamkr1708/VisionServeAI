# PHASE 1 — TRAINING SPECIFICATION
## VisionServe AI | Domain Plugin: NIH ChestXray14 | v1.0

**Status:** Draft for implementation
**Depends on:** `MASTER_ARCHITECTURE.md` (Phase 0, frozen)
**Produces:** The blueprint for the Kaggle training notebook
**Out of scope:** FastAPI, React, Docker, Triton, TensorRT, Redis, PostgreSQL, cloud, monitoring, deployment (all later phases)

This document is a design specification, not a tutorial. Every decision below is intentionally narrow — one option chosen, alternatives logged, switch criteria defined where relevant — so that implementation requires translation, not decision-making.

---

## 1. Training Objectives

**Primary objective.** Produce one correctly-trained, correctly-evaluated, correctly-exported baseline image classifier for the NIH ChestXray14 domain plugin, using a transfer-learning pipeline that is reproducible end-to-end from raw dataset to a validated ONNX artifact.

**Secondary objectives (educational).** The pipeline must give hands-on, defensible experience with:
- Transfer learning and progressive unfreezing on a real pretrained CNN
- Medical-image-appropriate preprocessing and augmentation
- Multi-label classification (as opposed to the more commonly tutorialized multi-class case)
- MLflow as a system of record for experiments, not just a logger
- PyTorch → ONNX export with numerical parity validation
- Disciplined handling of class imbalance and patient-level data leakage

**Explicit non-goals.**
- Beating published ChestXray14 leaderboard numbers
- Multi-view fusion, segmentation, or bounding-box regression
- Any inference-serving concern (covered in later phases per Phase 0)
- Any cloud/MLOps infra beyond MLflow tracking itself

**Definition of done for Phase 1.** Phase 1 succeeds when a documented, leakage-free, reproducible pipeline produces one validated baseline model and export bundle — not when a metric threshold is hit. Metric quality is tracked (Section 11) but Phase 2 is not gated on beating any external leaderboard.

## 2. Problem Definition

| Aspect | Decision |
|---|---|
| Task type | **Multi-label binary classification**, 14 independent pathology labels per image |
| Output head | 14 sigmoid outputs (independent probabilities), **not** softmax |
| Input | Single frontal/lateral chest radiograph, grayscale, replicated to 3 channels |
| "No Finding" | **Not** trained as a 15th class. Derived at inference as `1 - max(pathology probs > threshold)`. See rationale below. |
| Label cardinality | An image may have 0 (No Finding), 1, or multiple simultaneous pathology labels |

**Why sigmoid, not softmax.** Pathologies co-occur (e.g., Effusion + Atelectasis is common). Softmax would force the 14 classes to compete for a probability budget that sums to 1, which is the wrong inductive bias for a condition where multiple things can simultaneously be true. Each class gets its own independent BCE objective (Section 9).

**Why "No Finding" is not a trained class.** "No Finding" is logically the negation of all 14 pathologies, not an independent visual pattern. Training it as its own class creates a redundant, perfectly-correlated 15th target that adds no signal and can quietly dominate loss given its ~60% prevalence (Section 3). It is computed downstream from the 14 pathology decisions instead.

**Unit of prediction.** One image → one 14-dim probability vector. No patient-level aggregation in v1 (a patient may contribute several independent images/studies; each is scored independently).

## 3. Dataset Specification — NIH ChestXray14

### 3.1 Source structure
- 112,120 frontal-view chest X-ray PNGs from 30,805 unique patients, released by the NIH Clinical Center.
- Images are already downsampled by NIH to 1024×1024, 8-bit, grayscale, single channel.
- Metadata file: `Data_Entry_2017.csv` with columns: `Image Index`, `Finding Labels` (pipe-delimited, e.g. `Cardiomegaly|Effusion`), `Follow-up #`, `Patient ID`, `Patient Age`, `Patient Gender`, `View Position` (`PA`/`AP`), `OriginalImage[Width,Height]`, `OriginalImagePixelSpacing[x,y]`.
- 14 pathology labels: `Atelectasis, Cardiomegaly, Consolidation, Edema, Effusion, Emphysema, Fibrosis, Hernia, Infiltration, Mass, Nodule, Pleural_Thickening, Pneumonia, Pneumothorax`, plus the literal string `No Finding`.
- Labels were generated by **NLP mining of free-text radiology reports**, not direct radiologist adjudication of every image. NIH's own estimate puts label accuracy around ~90%. This is a known, accepted source of label noise — not something Phase 1 attempts to fix, but something the evaluation narrative (Section 11, 15) must acknowledge.
- `BBox_List_2017.csv` provides ~984 hand-verified bounding boxes covering only 8 of the 14 classes, intended for **localization evaluation**, not classification training. **v1 decision: bounding boxes are not used for training or loss in Phase 1.** They are reserved as a future qualitative check (does GradCAM activation fall inside the radiologist-drawn box for the subset where boxes exist?) — see Section 11 failure analysis.

### 3.2 Official split files
NIH ships `train_val_list.txt` (~86,524 images) and `test_list.txt` (~25,596 images), already constructed to be **patient-disjoint** between train_val and test.

**Decision:** Use the official `test_list.txt` as the held-out test set, untouched until final evaluation. Carve `train` and `val` out of `train_val_list.txt` ourselves, because NIH does not provide that split.

### 3.3 Internal train/val split — leakage prevention
**Hard rule:** a `Patient ID` must appear in exactly one of {train, val, test}. Never split by `Image Index` alone — many patients have multiple follow-up studies, and image-level random splitting silently leaks near-duplicate images of the same patient across splits, inflating validation metrics.

**Procedure:**
1. Build the unique list of `Patient ID`s present in `train_val_list.txt`.
2. Stratify patients (not images) by an approximate label-prevalence signature (e.g., does this patient ever have each of the 14 pathologies across their images) so rare classes aren't accidentally concentrated entirely in one split.
3. Split patients 80/20 into train/val.
4. Assign every image belonging to a patient to that patient's split.
5. Assert zero `Patient ID` intersection between the three final sets. This assertion is a **mandatory, automated, non-skippable check** before training starts — not a one-time manual eyeball.

### 3.4 Missing labels
No image in `Data_Entry_2017.csv` has a missing `Finding Labels` field — every row has either `No Finding` or at least one pathology. **No label imputation is needed in v1.** What does need handling is missing or corrupt *image files* (see 3.6).

### 3.5 Class imbalance (document, don't fix at the dataset level yet)
Approximate prevalence (per-image, multi-label so rows don't sum to 100%):

| Class | Approx. prevalence | Notes |
|---|---|---|
| No Finding | ~54% | Majority "class," excluded from training target (3.0) |
| Infiltration | ~17.7% | Most common pathology |
| Effusion | ~11.9% | |
| Atelectasis | ~10.3% | |
| Nodule | ~5.6% | |
| Mass | ~5.2% | |
| Pneumothorax | ~4.7% | |
| Consolidation | ~4.2% | |
| Pleural_Thickening | ~3.0% | |
| Cardiomegaly | ~2.5% | |
| Emphysema | ~2.2% | |
| Edema | ~2.0% | |
| Fibrosis | ~1.5% | |
| Pneumonia | ~1.3% | |
| **Hernia** | **~0.2%** | Extreme rarity, ~110–230 positive images total |

This long tail is the single biggest modeling risk in this project and is addressed at the loss level (Section 9), not by undersampling the majority or duplicating the minority at the dataset level — undersampling discards already-scarce data, and naive oversampling of ~200 Hernia images risks memorizing a handful of exact images rather than learning a generalizable pattern.

### 3.6 Image integrity checks (run once, before training)
- Verify every file referenced in the metadata CSV exists on disk.
- Open every image with `PIL.Image.verify()` to detect corrupt/truncated PNGs.
- Flag any image whose decoded size deviates from the `OriginalImage[Width,Height]` metadata by more than a sanity tolerance.
- **Policy:** corrupt or unreadable images are dropped and logged to a `excluded_images.csv` artifact with a reason code. They are never imputed or replaced. Given dataset size (112K images), losing a handful of corrupt files has no measurable effect on training.

### 3.7 Known data quality caveat to carry into EDA
A well-documented quirk of this dataset: a small number of rows report implausible `Patient Age` values (e.g., >100, including some encoded as 3-digit values like 150). This is **not** an image corruption issue, it's a metadata entry issue. Decision: keep these rows for image-label training (age is not a model input), but flag/exclude them specifically from any age-based EDA aggregate so the age histogram isn't distorted (Section 4).

## 4. EDA Plan

EDA here is *gatekeeping*, not exploration for its own sake — every item below exists to catch a problem before training, or to set a number Phase 1 explicitly needs (a threshold, a weight, a split sanity check).

### 4.1 Required statistics
1. Total image count, total unique patient count, images-per-patient distribution (min/median/max).
2. Per-class positive count and prevalence %, computed separately for train, val, and test — used to confirm the patient-wise split (3.3) didn't accidentally skew rare-class prevalence between splits.
3. Label cardinality distribution: histogram of "number of pathology labels per image" (0, 1, 2, 3+).
4. Pairwise label co-occurrence matrix (14×14), since this is multi-label and single-label frequency alone hides clinically expected co-occurrence (e.g. Effusion↔Atelectasis).
5. View position split (PA vs AP) — AP views are disproportionately taken in sicker/bedridden patients, which is a known confound worth knowing about, not worth correcting for in v1.
6. Gender distribution.
7. Age distribution **after excluding the implausible values identified in 3.7**, plus a separate count of how many rows were excluded and why.
8. Image dimension consistency check across the full dataset (expect ~all 1024×1024, but verify rather than assume).
9. Global pixel-intensity histogram (sampled subset, not all 112K images) to sanity-check exposure/contrast range before normalization decisions are finalized (Section 5).

### 4.2 Required plots
- Horizontal bar chart: per-class prevalence (train/val/test overlaid), sorted descending.
- Heatmap: 14×14 co-occurrence matrix.
- Histogram: label cardinality per image.
- Histogram: patient age (post-cleaning).
- Bar chart: gender and view-position counts.
- A sample image grid: a handful of example images per class (positive examples), purely for visual sanity-checking that labels look plausible — this is a qualitative spot check, not a metric.

### 4.3 Explicitly out of scope for this EDA
- No attempt to validate label correctness against radiologist ground truth beyond what NIH already provides (3.1's noise caveat is accepted, not investigated further in Phase 1).
- No deep-dive EDA on the bounding-box subset — one summary line (how many boxes, which 8 classes) is sufficient since it isn't used for training.
- No severity/grading analysis (the dataset has no severity labels to analyze).
- No pixel-spacing/physical-size analysis — irrelevant to a 2D classification baseline.

**Deliverable:** one EDA artifact (notebook section + exported plots) logged to MLflow as part of the eventual run, per Section 13.

## 5. Image Preprocessing

| Step | Decision | Rationale |
|---|---|---|
| Resize | 224×224, direct resize (no letterboxing/padding) | Source images are already near-square (1024×1024); chest anatomy framing is consistent enough across the dataset that aspect-distorting resize is standard practice here and matches CheXNet-era precedent |
| Interpolation | Bilinear | Standard, cheap, sufficient at this downsampling ratio |
| Color space | Grayscale → replicated to 3 identical channels | Backbone is ImageNet-pretrained on RGB; replication lets pretrained filters apply without architecture surgery on the first conv layer |
| Normalization | ImageNet mean/std (`[0.485,0.456,0.406]` / `[0.229,0.224,0.225]`) | Matches the pretrained backbone's input distribution; computing dataset-specific mean/std is a defensible alternative but adds a moving part for negligible benefit when channels are a grayscale replication anyway |
| Bit depth | Decode as 8-bit, scale to [0,1] before normalization | Standard `ToTensor()` behavior |
| Quality checks | Per Section 3.6, run once at dataset-prep time, not per-batch at train time | Avoids paying integrity-check cost every epoch |
| Corrupt image handling at load time | If a listed image still fails to decode at `Dataset.__getitem__` time (defensive, should be near-zero after 3.6), skip and log — **never** substitute a blank/zero tensor, which would silently inject a fake negative-looking sample | Prevents a corrupt file from becoming invisible training noise |

**Decision explicitly deferred, not adopted in v1:** CLAHE / histogram equalization as a preprocessing step. It's a reasonable medical-imaging technique but it changes the input distribution in a way that should be evaluated as an *augmentation experiment* (Section 6 optional list) against a clean baseline, not baked into preprocessing where its effect can't be isolated.

## 6. Augmentation Strategy

**Governing principle:** augmentation must not create an anatomically implausible image or alter a feature the label depends on. Chest radiographs are not natural images — generic vision augmentation recipes (heavy color jitter, mixup, elastic deformation) carry real risk of corrupting the very signal the model is supposed to learn.

### 6.1 Validation and test sets
**No stochastic augmentation.** Only the deterministic preprocessing in Section 5 (resize + normalize). Validation and test transforms must be identical and fixed for every run, so metrics are comparable across experiments.

### 6.2 Training set — mandatory
| Augmentation | Parameters | Why mandatory |
|---|---|---|
| Random rotation | ±10° | Patient positioning varies slightly study to study; small rotation is realistic and doesn't alter pathology appearance |
| Brightness/contrast jitter | Small range (e.g. ±10%) | Simulates real exposure/technique variation across different X-ray machines and settings in the source hospitals |

### 6.3 Training set — optional / explicitly deferred
| Augmentation | Status | Switch criterion |
|---|---|---|
| Random crop/zoom (small scale, e.g. 0.9–1.0) | Deferred | Consider if overfitting is observed (train/val AUROC gap widening) after baseline run |
| CLAHE contrast enhancement | Deferred | Consider as a controlled A/B experiment against the baseline, not a default |
| Gaussian noise injection | Deferred | Only if model appears to be relying on sensor-specific artifacts rather than anatomy (would show up in GradCAM failure analysis, Section 11) |
| Cutout | Deferred | Low priority; benefit on this dataset is unproven and risks occluding small findings (nodules) |
| Mixup / CutMix | Deferred | Multi-label mixup requires soft-label loss handling that adds real complexity; not worth introducing before the single-label-equivalent pipeline is proven |

### 6.4 Explicitly forbidden in this domain
- **Vertical flip** — anatomically invalid orientation for a chest X-ray, never realistic.
- **Aggressive hue/saturation jitter** — meaningless on a grayscale-replicated image and a sign the augmentation library's default natural-image recipe wasn't adapted for this domain.
- **Elastic/grid deformation** — risks fabricating geometry that looks pathological (e.g., simulating a mass or distorting a heart border) where none exists; unacceptable for a diagnostic-style label.

### 6.5 Horizontal flip — flagged for explicit decision, not silently included
Horizontal flip is **enabled** in v1 (probability 0.5) on the grounds that overall pathology classification (not laterality-specific localization) is the v1 task, and the published literature on this dataset commonly uses it. **However:** this is recorded here as a flagged, debatable choice because a small number of findings (e.g., side-specific Pneumothorax) are technically laterality-sensitive. If Section 11's per-class failure analysis shows laterality-relevant classes underperforming in a pattern consistent with flip confusion, horizontal flip should be the first augmentation removed in a follow-up experiment — not silently kept.

### 6.6 Test-Time Augmentation (TTA)
**Not implemented in v1.** Flagged as a Phase 2+ enhancement (horizontal-flip TTA averaging is the natural first candidate given 6.5). No TTA code path needs to exist in the Phase 1 notebook.

## 7. Backbone Evaluation & Selection

### 7.1 Candidates considered

| Backbone | Params (approx) | Pretrained availability | Precedent on this dataset | ONNX export maturity | Fit for free-tier GPU |
|---|---|---|---|---|---|
| **DenseNet121** | ~7.9M | Yes (torchvision) | Strong — this is the CheXNet architecture, the most-cited baseline specifically on ChestXray14 | Mature, widely exported | Good — small enough to fine-tune fully on a T4/P100 within a Kaggle session |
| ResNet50 | ~25.6M | Yes (torchvision) | Common general baseline, used in some ChestXray14 papers | Mature | Good, but 3× the params for no clear accuracy edge on this task |
| EfficientNet-B0 | ~5.3M | Yes (timm/torchvision) | Less established specifically on this dataset | Good, occasionally fiddly opsets (depthwise/SE blocks) | Good |
| ViT-Tiny/Small | ~5–22M | Yes (timm) | Weak precedent on ChestXray14; transformers generally need more data or longer schedules to match CNN inductive bias at this dataset scale | Workable but historically more export friction (attention ops, dynamic shapes) | Adequate, but higher risk for a first export-pipeline exercise |
| ConvNeXt-Tiny | ~28M | Yes (timm) | Minimal precedent on this dataset | Good | Heavier than needed for a first baseline |

### 7.2 Decision: **DenseNet121**

**Why:**
1. **Direct literature comparability.** DenseNet121 is the backbone used by CheXNet, the canonical baseline on this exact dataset. Reproducing a known architecture on a known dataset gives a meaningful sanity check ("are my numbers in a plausible range") that a less-precedented backbone wouldn't.
2. **Right-sized, not smallest or biggest.** At ~7.9M params it is comfortably trainable end-to-end (not just head fine-tuning) within free Kaggle/Colab GPU session limits and time budgets.
3. **Architectural fit.** Dense connectivity reuses low-level texture features across the network depth, which suits the subtle, texture-and-density-based nature of many of these findings (e.g., Infiltration, Fibrosis) better than discarding low-level features the way deeper purely-sequential ResNets tend to.
4. **Export maturity.** DenseNet's operator set (conv, batchnorm, relu, concat, pooling) is thoroughly supported by ONNX opset and ONNXRuntime — no esoteric ops to debug in Section 14's parity check, which matters because this project also exists to learn the export pipeline cleanly.

**Why not the others, explicitly:**
- *Not ResNet50* — reasonable alternative, but 3× the parameter count with no established accuracy advantage on this task; doesn't serve the "optimize for learning and deployment, not complexity" principle in Phase 0.
- *Not EfficientNet-B0* — viable, but lacks the direct precedent that makes result-sanity-checking easy for a first baseline; flagged as a good **second backbone to compare against** once the pipeline itself is proven.
- *Not ViT variants* — higher risk on two axes at once (less proven on this dataset *and* historically more export friction) for a project whose immediate goal is proving the whole pipeline, not maximizing architecture novelty.
- *Not ConvNeXt-Tiny* — heaviest of the credible options with the least specific justification for this dataset size and free-tier compute budget.

**Note for the backbone-evaluation deliverable:** Phase 1 implementation should still empirically confirm pretrained-weight availability and a basic forward pass for DenseNet121 (and ideally EfficientNet-B0 as a documented "candidate #2" smoke test) before committing — this is a 10-minute check, not a full second training run.

## 8. Transfer Learning Strategy

### 8.1 Staged unfreezing schedule

| Stage | Epoch range | Trainable layers | Backbone LR | Head LR | Rationale |
|---|---|---|---|---|---|
| Stage 1 | 1–3 | Classifier head only; all DenseNet121 conv blocks (denseblock1–4) frozen | 0 (frozen) | `3e-4` | Let the new 14-way head adapt to ImageNet features before any backbone weight is disturbed — avoids destroying pretrained filters with large early gradients from a randomly-initialized head |
| Stage 2 | 4–8 | Unfreeze denseblock4 + transition3; denseblock1–3 stay frozen | `3e-5` | `3e-4` | Adapt the deepest, most task-specific features to chest-X-ray texture statistics while preserving generic low/mid-level edge and texture filters |
| Stage 3 | 9+ (until early stopping, Section 10) | Unfreeze everything | `1e-5` | `1e-4` | Full fine-tune at low LR once the head and late layers have stabilized; only proceed here if Stage 2 validation AUROC is still improving — if Stage 2 has plateaued, Stage 3 may be skipped rather than risk overfitting the long tail classes |

**Differential LR rule:** whenever backbone layers are trainable, backbone LR = head LR / 10. The head was randomly initialized and needs to move faster; the backbone was already pretrained and needs only gentle adaptation.

### 8.2 Why progressive unfreezing instead of either extreme
- **Not full-frozen-forever (pure linear-probe):** a frozen DenseNet121 trained only as a generic feature extractor under-fits the X-ray-specific texture patterns that ImageNet never exposed it to (chest X-rays are visually nothing like ImageNet's natural-image distribution).
- **Not full-unfreeze-from-epoch-1:** with a relatively small, noisy-labeled, imbalanced dataset, unfreezing everything immediately risks catastrophic forgetting of useful pretrained low-level filters before the head has even learned a sensible decision boundary — the large initial loss from the random head would propagate large, disruptive gradients into early conv layers.

### 8.3 Stage transition criterion
Move to the next stage only when val macro AUROC improvement over the last 2 epochs falls below a small delta (e.g. `0.002`), **or** the stage's max epoch is reached — whichever comes first. This is a measured trigger, not a fixed epoch count alone, but the fixed ranges above serve as the default/fallback so the schedule is still fully specified if no human is watching the run live (important given Kaggle/Colab session time limits).

## 9. Loss Function Strategy

### 9.1 Baseline (v1, mandatory)
**`BCEWithLogitsLoss`, applied independently per class, with per-class `pos_weight`.**

- This is the textbook-correct loss for multi-label classification with independent binary targets (Section 2) — one binary cross-entropy term per class, summed/averaged across the 14 outputs.
- `pos_weight` per class = `(negative_count / positive_count)` computed from the **training split only** (never from val/test, to avoid leaking split statistics into the loss). This directly counteracts the prevalence imbalance documented in Section 3.5 — Hernia's loss contribution per positive example is weighted far higher than No-Finding-dominant classes', without altering what data the model actually sees (unlike resampling).
- `pos_weight` is computed once from the training set and frozen for the run; it is a logged hyperparameter (Section 13), not something recomputed per batch.

### 9.2 Why not start with something fancier
Focal Loss, class-balanced loss, and Asymmetric Loss (ASL) are all legitimate, more sophisticated answers to imbalance — but each adds at least one extra tunable hyperparameter (focusing parameter γ, balancing β, asymmetric margins) on top of a pipeline that hasn't yet been validated end-to-end. Introducing tuning complexity before the simpler baseline's behavior is understood and logged risks confusing "the loss function is wrong" with "the pipeline has a bug" when results look off.

### 9.3 Documented future alternatives and switch criteria

| Alternative | Consider switching when... |
|---|---|
| **Focal Loss** | Weighted BCE still leaves the rarest classes (Hernia, Fibrosis, Pneumonia) with near-zero recall after threshold optimization (Section 11) despite `pos_weight` |
| **Class-Balanced Loss** (Cui et al., effective-number-of-samples reweighting) | `pos_weight`'s simple inverse-frequency weighting is found to overcorrect or destabilize training (loss spikes, the rarest classes' logits diverging) |
| **Asymmetric Loss (ASL)** | Project moves toward matching modern multi-label SOTA benchmarks specifically, beyond the "prove the pipeline works" goal of Phase 1 |

### 9.4 Implementation contract
The loss module must be a single configurable component (`loss_type: "weighted_bce"` default) so a future swap is a config change, not a rewrite — but **only the weighted-BCE path needs to be implemented in Phase 1.** Do not pre-build Focal/CB/ASL implementations now; build the abstraction point, not the alternatives.

## 10. Optimizer, Scheduler & Training Mechanics

| Mechanism | Decision | Justification |
|---|---|---|
| **Optimizer** | AdamW, `weight_decay ≈ 1e-4`, decay excluded on bias and BatchNorm parameters | AdamW's decoupled weight decay is the standard modern default for fine-tuning pretrained CNNs; excluding norm/bias params from decay is standard practice that avoids penalizing parameters that aren't meant to shrink toward zero |
| **LR schedule** | Cosine annealing with linear warmup | Predictable, low-hyperparameter-sensitivity shape that behaves well even if a Kaggle/Colab session ends early and training resumes (Section 12) — preferred here over OneCycle specifically because OneCycle's shape is more sensitive to getting `total_steps` exactly right, which is fragile across resumed/interrupted free-tier sessions |
| **Warmup** | Linear warmup over the first 5% of total optimizer steps | Prevents early instability, particularly at each unfreezing transition (Section 8) where newly-trainable layers start receiving gradients again |
| **Gradient accumulation** | Configurable, default OFF; turn on only if the desired effective batch size (target: 32–64) doesn't fit in T4/P100 VRAM at 224×224 | Free-tier GPUs (Kaggle T4 ×1 16GB, Colab T4/P100) can usually fit this batch size at this resolution directly; accumulation is documented as a config knob, not assumed necessary |
| **Mixed precision** | **Mandatory: AMP (fp16) via `torch.cuda.amp`** | Meaningfully speeds up training and reduces memory pressure on exactly the free-tier GPUs this project targets — this isn't optional given the hardware constraint in Phase 0 |
| **Early stopping** | Monitor val macro AUROC; patience = 5 epochs; `min_delta = 0.001` | Protects limited free-GPU compute hours from being spent on a plateaued run; macro AUROC chosen as the monitored metric to match the primary evaluation metric (Section 11) |
| **EMA (exponential moving average of weights)** | **Optional, OFF by default in v1**; config flag present but not required to be used | Genuinely useful technique, but adds a second set of weights to track, checkpoint, and reason about — deferred so the core pipeline is validated first; can be turned on in a follow-up experiment without architecture changes |

## 11. Evaluation Protocol

### 11.1 Primary metric
**Macro-averaged AUROC across the 14 pathology classes.** Chosen because it is (a) threshold-independent, (b) the standard headline metric in ChestXray14 literature, enabling sanity comparison, and (c) robust to imbalance in the sense that each class contributes equally to the average regardless of how rare it is — which is exactly the property needed to keep Hernia-level rarity from being invisible in an aggregate number.

### 11.2 Secondary metrics
- **Micro-averaged AUROC** — reported alongside macro to make the imbalance's effect visible (a large micro/macro gap signals the model is doing well on common classes and poorly on rare ones).
- **Per-class AUROC** — full 14-row table, every run, every time. Never report only the aggregate.
- **Per-class Average Precision (AP)** — more informative than AUROC specifically for the rarest classes, since AUROC can look deceptively good on a highly imbalanced class even when precision at any usable threshold is poor.

### 11.3 Threshold optimization
Multi-label classification has **no single correct 0.5 cutoff** across 14 classes with wildly different prevalence. Per-class thresholds are optimized on the **validation set only** (test set stays untouched until final reporting) by maximizing per-class F1 (or equivalently, Youden's J on the ROC curve — F1-maximization is preferred here because it more directly reflects the precision/recall tradeoff a clinical-style output would need). The resulting 14 thresholds are a **frozen, versioned artifact** (`thresholds.json`, Section 14), not a runtime-recomputed value.

### 11.4 Confusion matrices
One **binary** confusion matrix per class (positive vs. negative for that pathology, at its optimized threshold) — not a single 14-class matrix, which doesn't meaningfully exist in a multi-label setting.

### 11.5 ROC and PR curves
Generated and logged per class (14 ROC curves, 14 PR curves) as MLflow artifacts (Section 13). PR curves are specifically called out as important to actually look at for the bottom 3–4 rarest classes, where AUROC alone can be misleadingly high.

### 11.6 Failure analysis
- **GradCAM** on a sample of false positives and false negatives **per class**, with mandatory extra attention on the three rarest classes (Hernia, Fibrosis, Pneumonia) where small absolute error counts can hide large relative failure rates.
- For the subset of images that have NIH bounding-box annotations (3.1) and fall into one of the 8 boxed classes: a qualitative check of whether GradCAM activation overlaps the radiologist-drawn box. This is **qualitative/spot-check only** in v1, not a quantitative localization metric — formal IoU-style localization scoring is out of scope for a classification-only Phase 1.
- A short written note acknowledging the ~10% label-noise ceiling (Section 3.1): when failure analysis turns up a confident, visually-plausible model prediction that disagrees with the NLP-derived label, that should be recorded as a candidate label-noise case, not automatically treated as a model error.

## 12. Checkpoint Strategy

**Naming convention:**
```
{backbone}_{stage}_epoch{NN}_valauroc{X.XXX}.pt
e.g. densenet121_stage2_epoch07_valauroc0.812.pt
```

**Retention policy:**
- **Best checkpoint:** the single highest val-macro-AUROC checkpoint seen so far, overwritten whenever a new best is found. Tagged `*_best.pt`.
- **Last checkpoint:** the most recent epoch's checkpoint, always overwritten each epoch. Tagged `*_last.pt`.
- Only these two are retained long-term (not every epoch) — Kaggle/Colab free-tier disk and output quotas are limited, and intermediate non-best checkpoints have no use case in this project.

**Checkpoint contents (every save, not just final):**
- Model `state_dict`
- Optimizer `state_dict`
- Scheduler `state_dict`
- Current epoch number and current stage (Section 8)
- Best val macro AUROC so far
- RNG state (Python/NumPy/Torch/CUDA seeds)

**Why full optimizer/scheduler/RNG state, not just weights:** Kaggle sessions cap out at fixed runtime limits and Colab free tier disconnects unpredictably. A checkpoint that only stores weights forces a cold-restart of momentum and LR schedule on resume, which silently changes the effective training trajectory. Storing full state makes resume numerically equivalent to uninterrupted training.

**Storage location:** checkpoints must be written to persistent storage outside the ephemeral session filesystem (Kaggle "Output" dataset, or a mounted Google Drive in Colab) — never left only in `/kaggle/working` or local Colab runtime storage, both of which are deleted when the session ends.

## 13. Experiment Tracking (MLflow)

Every training run is one MLflow run. Nothing below is optional logging — this is the contract for what "a complete run" means in this project.

### 13.1 Parameters (logged once, at run start)
- Backbone name, image input size, batch size, effective batch size (if gradient accumulation used)
- Optimizer name + LR (head and backbone), weight decay, scheduler type, warmup fraction
- Loss type, per-class `pos_weight` values
- Augmentation config (every flag/parameter from Section 6, including which optional augmentations were on)
- Stage boundaries (epoch ranges from Section 8) and which stages actually ran
- Random seed
- Dataset split file identities (a hash of `train_ids.txt` / `val_ids.txt` / `test_ids.txt`, so a run can later be proven to have used a particular split version)

### 13.2 Metrics (logged every epoch)
- Train loss, val loss
- Val macro AUROC, val micro AUROC
- Per-class val AUROC (logged as 14 distinct metric keys, e.g. `auroc_hernia`, `auroc_cardiomegaly`, …) so per-class curves are queryable over training time, not just at the final epoch
- Current learning rate (head and backbone) — makes the warmup/cosine/stage-transition schedule visible after the fact without re-deriving it from config

### 13.3 Artifacts (logged at run end, or at best-checkpoint time)
- Best and last model checkpoints (Section 12)
- Per-class confusion matrices (as images)
- Per-class ROC curves and PR curves (as images)
- GradCAM sample grids from failure analysis (Section 11.6)
- Training curve plots: loss vs. epoch, macro AUROC vs. epoch
- `thresholds.json` (per-class optimized decision thresholds, Section 11.3)
- EDA summary artifact (Section 4 deliverable) — linked/logged once per dataset-version, reusable across runs against the same split
- Environment snapshot (`pip freeze` / `requirements.txt`) for reproducibility

### 13.4 Run tagging
Tag every run with: dataset version identifier, split version identifier, and a notebook/commit version string. This is what makes "which exact data and code produced this checkpoint" answerable six months later.

## 14. Model Export Contract

**Pipeline:** PyTorch → ONNX → Validation → Model Registry. A model only reaches "Model Registry" status if every artifact below exists and the validation step passes.

### 14.1 Required artifact bundle (one versioned folder per promoted model)

| # | Artifact | Purpose |
|---|---|---|
| 1 | `model.pt` | PyTorch `state_dict` + architecture config, for any future PyTorch-side re-evaluation |
| 2 | `model.onnx` | Exported graph; **opset pinned** (e.g. opset 17) so behavior doesn't drift with ONNX/ONNXRuntime version upgrades; dynamic batch axis, fixed spatial input size (224×224 per Section 5) |
| 3 | `export_validation_report.json` | Numerical parity check results (14.2) |
| 4 | `thresholds.json` | Per-class decision thresholds from Section 11.3 — required because raw sigmoid outputs are meaningless without them |
| 5 | `preprocessing_config.json` | Exact resize size, interpolation method, normalization mean/std, channel handling — frozen, must be reproduced byte-for-byte at any future inference time |
| 6 | `labels.json` | Ordered index→class-name mapping for the 14 outputs. **This ordering must never silently change between versions** — it's the single easiest way to silently corrupt downstream inference |
| 7 | `model_card.md` | Class list, the full per-class metrics table (Section 11), known limitations (label noise ceiling, imbalance, single-view-only, no localization), intended use statement, training data summary, and the dataset's license/usage terms |

### 14.2 Export validation procedure (mandatory gate)
1. Export `model.pt` → `model.onnx` with the pinned opset and dynamic batch axis.
2. Run `onnx.checker.check_model()` — must pass with no errors.
3. Run inference with ONNXRuntime on a fixed, held-out batch of validation images.
4. Run the same batch through the original PyTorch model in eval mode.
5. Compute max absolute difference between the two outputs, per class, per sample.
6. **Gate:** max abs diff must be below a defined tolerance (e.g. `1e-4`). If it fails, the export is rejected and the discrepancy must be root-caused (common causes: batchnorm eval-mode mismatch, dropout not disabled, mismatched preprocessing between the two paths) before re-attempting — **never** silently raise the tolerance to make a failing export pass.
7. Log the full parity report (per-class max/mean diff) as `export_validation_report.json` and as an MLflow artifact (Section 13.3).

### 14.3 Promotion criterion
A model is only registered/promoted if:
- Export validation (14.2) passes
- All 7 artifacts in 14.1 are present
- Val macro AUROC meets the minimum sanity bar defined in Section 15 (the acceptance criterion, not a leaderboard target)

## 15. Acceptance Criteria — Phase 1 Exit Checklist

Phase 2 does not begin until every item below is checked. These are gates, not aspirations.

- [ ] Patient-wise train/val/test split constructed; zero `Patient ID` overlap verified by an automated assertion (Section 3.3); split file hashes recorded.
- [ ] Image integrity check completed; `excluded_images.csv` produced (even if empty) (Section 3.6).
- [ ] EDA report produced with all required statistics and plots (Section 4), logged to MLflow.
- [ ] Preprocessing and augmentation pipeline implemented exactly per Sections 5–6, with val/test transforms verified deterministic (no stochastic ops active outside training).
- [ ] DenseNet121 baseline trained through the staged unfreezing schedule (Section 8), with stage transitions logged.
- [ ] Loss function implemented as configurable weighted-BCE per Section 9, with `pos_weight` values computed from train-split-only and logged.
- [ ] Training run completed with AMP enabled, cosine+warmup schedule, and early stopping per Section 10.
- [ ] Full MLflow run present with **every** parameter, metric, and artifact category from Section 13 — partial logging does not satisfy this item.
- [ ] Best checkpoint achieves a val macro AUROC **at or above 0.75** — chosen as a sanity bar confirming the pipeline is learning correctly (well below published SOTA on purpose; this is a "did the pipeline work" gate, not a leaderboard target per Section 1's non-goals).
- [ ] Per-class thresholds computed on validation set and exported as `thresholds.json` (Section 11.3).
- [ ] Full per-class metrics table (AUROC, AP, confusion matrix, ROC/PR curves) produced and logged (Section 11).
- [ ] Failure analysis / GradCAM review completed, with mandatory coverage of the three rarest classes (Hernia, Fibrosis, Pneumonia) (Section 11.6).
- [ ] ONNX export completed and passes the numerical parity gate in Section 14.2 (max abs diff below tolerance, logged in `export_validation_report.json`).
- [ ] Full 7-artifact export bundle (Section 14.1) present and versioned, including a completed `model_card.md`.
- [ ] Checkpoint and export artifacts confirmed written to persistent (non-ephemeral) storage (Section 12).

When every box above is checked, Phase 1 is complete and Phase 2 may begin against a known-good, fully-documented baseline.
